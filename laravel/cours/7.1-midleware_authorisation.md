
# CHAPITRE 7

## Middlewares, Contrôleurs et Sécurisation des Accès (Laravel 12)

---

## 1. Introduction

À ce stade du projet fil rouge, l’application contient :

* plusieurs contrôleurs (Service, Reservation, Medecin, Admin…)
* plusieurs routes
* plusieurs rôles (admin, médecin, patient)

Cependant, un problème fondamental subsiste :

> Les contrôleurs ne doivent jamais décider seuls qui a le droit d’accéder à une action.

La sécurité **ne se met pas dans les contrôleurs**, mais **avant** qu’ils ne soient appelés.

C’est exactement le rôle des **middlewares**.

---

## 2. Problème réel à résoudre

Sans middleware :

* un patient peut appeler un contrôleur admin
* un utilisateur non connecté peut créer une réservation
* `auth()->user()` peut être `null`
* les contrôleurs semblent “ne pas fonctionner”

👉 Le problème n’est **pas le contrôleur**,
👉 le problème est **le chemin d’accès jusqu’au contrôleur**.

---

## 3. Rappel : chemin d’une requête Laravel

Lorsqu’un utilisateur fait une requête :

```
Navigateur
   ↓
Route
   ↓
Middleware(s)
   ↓
Contrôleur
   ↓
Vue / Réponse
```

Le contrôleur **n’est exécuté que si les middlewares autorisent l’accès**.

---

## 4. Différence entre authentification et autorisation

### 4.1 Authentification

Question :

> L’utilisateur est-il connecté ?

Laravel fournit déjà le middleware :

```
auth
```

---

### 4.2 Autorisation

Question :

> L’utilisateur a-t-il le droit d’accéder à cette action ?

Dans notre projet, cela dépend du **rôle** :

* admin
* medecin
* patient

👉 Cela nécessite un **middleware personnalisé**.

---

## 5. Pourquoi ne pas vérifier les rôles dans les contrôleurs

Exemple à éviter :

```php
if (auth()->user()->role !== 'admin') {
    abort(403);
}
```

Pourquoi c’est une mauvaise pratique :

* duplication du code
* contrôleurs surchargés
* sécurité difficile à maintenir
* risque d’oubli sur certaines méthodes

La bonne pratique :

> Le contrôleur suppose que l’accès est déjà autorisé.

---

## 6. Création du middleware de rôle (Laravel 12)

### 6.1 Génération du middleware

```bash
php artisan make:middleware RoleMiddleware
```

Fichier créé :

```
app/Http/Middleware/RoleMiddleware.php
```

---

### 6.2 Code du middleware

```php
<?php

namespace App\Http\Middleware;

use Closure;
use Illuminate\Http\Request;

class RoleMiddleware
{
    public function handle(Request $request, Closure $next, string $role)
    {
        if (!auth()->check()) {
            return redirect('/login');
        }

        if (auth()->user()->role !== $role) {
            abort(403);
        }

        return $next($request);
    }
}
```

#### Explication :

* `auth()->check()` : vérifie si l’utilisateur est connecté
* comparaison du rôle : contrôle l’autorisation
* `abort(403)` : accès interdit
* `$next($request)` : autorise l’accès au contrôleur

---

## 7. Déclaration du middleware (IMPORTANT – Laravel 12)

En Laravel 12 :

* le fichier `Kernel.php` **n’est plus utilisé**
* les middlewares sont déclarés dans `bootstrap/app.php`

---

### 7.1 Fichier concerné

```
bootstrap/app.php
```

---

### 7.2 Déclaration correcte

Ajouter ou compléter :

```php
use App\Http\Middleware\RoleMiddleware;
use Illuminate\Foundation\Configuration\Middleware;

->withMiddleware(function (Middleware $middleware) {
    $middleware->alias([
        'role' => RoleMiddleware::class,
    ]);
})
```

Cette déclaration :

* associe le nom `role`
* au middleware `RoleMiddleware`
* rend possible l’utilisation `role:admin` dans les routes

---

## 8. Application du middleware aux routes

### 8.1 Routes administrateur

```php
Route::middleware(['auth', 'role:admin'])->group(function () {

    Route::get('/dashboard/admin', function () {
        return view('dashboard.admin');
    });

    Route::get('/admin/services', [AdminServiceController::class, 'index']);
    Route::get('/admin/users', [AdminUserController::class, 'index']);
    Route::get('/admin/reservations', [AdminReservationController::class, 'index']);

});
```

---

### 8.2 Routes médecin

```php
Route::middleware(['auth', 'role:medecin'])->group(function () {

    Route::get('/dashboard/medecin', function () {
        return view('dashboard.medecin');
    });

});
```

---

### 8.3 Routes patient

```php
Route::middleware(['auth', 'role:patient'])->group(function () {

    Route::get('/dashboard/patient', function () {
        return view('dashboard.patient');
    });

});
```

---

## 9. Conséquence directe sur les contrôleurs

Grâce aux middlewares :

* les contrôleurs reçoivent toujours un utilisateur valide
* `auth()->user()` fonctionne
* `auth()->id()` fonctionne
* aucun contrôle de rôle n’est nécessaire dans les contrôleurs

Exemple valide :

```php
$validated['user_id'] = auth()->user()->id;
```

---

## 10. Vérifier que les contrôleurs fonctionnent correctement

### 10.1 Vérifier la route

* bonne URL
* bonne méthode HTTP
* bon middleware

---

### 10.2 Vérifier l’authentification

Test temporaire :

```php
dd(auth()->user());
```

Résultat attendu :

* objet `User`
* jamais `null`

---

### 10.3 Vérifier l’appel du contrôleur

```php
dd('store atteint');
```

Si ce message ne s’affiche pas :

* la route ne pointe pas vers la méthode
* ou la méthode HTTP est incorrecte

---

### 10.4 Vérifier les données validées

```php
dd($validated);
```

Si les données sont correctes :

* le contrôleur fonctionne
* le problème est ailleurs (migration, modèle)

---

## 11. Erreurs fréquentes à éviter

* oublier de déclarer le middleware dans `bootstrap/app.php`
* confondre Laravel 10 et Laravel 12
* oublier `auth` dans les routes
* vérifier les rôles dans les contrôleurs
* appeler une route POST avec GET
