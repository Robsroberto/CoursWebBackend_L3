## 12.1 Objectif général

Au chapitre 10, tu as construit une API REST complète avec **Laravel + Sanctum**.

Dans ce chapitre, tu vas **réécrire le même backend** en utilisant **Node.js + Express.js**.

L'objectif est double :

* comprendre que les **concepts REST, JWT, rôles, middlewares** sont universels — ils s'appliquent dans n'importe quel langage ou framework
* découvrir l'écosystème **Node.js** qui est aujourd'hui le plus utilisé dans le monde du web

> Remarque : les deux backends partagent la **même base MySQL** (`reservation_app`). Les tables ont déjà été créées par Laravel. Express va juste les lire et les manipuler.

---

## 12.2 Qu'est-ce qu'Express.js ?

**Express.js** est le framework web minimaliste de Node.js.

Il fournit :

* un système de **routage** (GET, POST, PUT, DELETE)
* un mécanisme de **middlewares** (fonctions exécutées avant/après les routes)
* des utilitaires pour la gestion des **requêtes et réponses HTTP**

Il ne fournit PAS (contrairement à Laravel) :

* d'ORM intégré → on utilise **Sequelize** ou Prisma
* d'authentification intégrée → on implémente **JWT manuellement**
* de validation intégrée → on utilise **express-validator**
* de documentation API → on utilise **swagger-jsdoc**

> Remarque : Express est **volontairement minimaliste**. Tu construis exactement ce dont tu as besoin. C'est sa force et sa difficulté.

---

## 12.3 Différences clés : Laravel vs Express

| Aspect | Laravel (PHP) | Express (Node.js) |
|---|---|---|
| Langage | PHP | JavaScript |
| Paradigme | MVC complet, opinionné | Minimaliste, flexible |
| ORM | Eloquent (intégré) | Sequelize, Prisma (externe) |
| Auth | Sanctum (intégré) | JWT manuel (jsonwebtoken) |
| Validation | FormRequest intégré | express-validator (externe) |
| Routage | routes/api.php | routes/*.js (libre) |
| Démarrage | `php artisan serve` | `node server.js` |
| Doc API | Swagger externe | swagger-jsdoc (annotations) |
| Hot reload | artisan + php | nodemon |
| Port défaut | 8000 | 3000 |

> Les **concepts** sont identiques. Seule la **syntaxe et les outils** changent.

### 📁 `src/config/database.js`

```js
const { Sequelize } = require('sequelize');
require('dotenv').config();

const sequelize = new Sequelize(
  process.env.DB_NAME,
  process.env.DB_USER,
  process.env.DB_PASSWORD,
  {
    host: process.env.DB_HOST,
    port: parseInt(process.env.DB_PORT) || 3306,
    dialect: 'mysql',
    logging: false,
    define: {
      underscored: true,   // colonnes en snake_case (created_at, updated_at)
      timestamps: true,
    },
  }
);

module.exports = sequelize;
```

### Pourquoi `underscored: true` ?

Laravel crée des colonnes en **snake_case** : `created_at`, `updated_at`, `medecin_id`.
Sequelize par défaut utilise du camelCase. L'option `underscored: true` force le snake_case pour être **compatible avec les tables Laravel**.

## 12.8 Étape 3 – Les modèles Sequelize

### Principe clé : `sync: false` implicite

Avec Sequelize, on peut soit :
* laisser Sequelize **créer les tables** (`sync({ force: true })`) — dangereux en production
* décrire les tables **existantes** sans les modifier — c'est notre approche

On ne lancera **jamais** `sequelize.sync()`. Sequelize mappe simplement les objets JS sur les tables existantes.

---

```js
const User        = require('./User');

const Reservation = require('./Reservation');

// Un médecin possède plusieurs services
User.hasMany(Service,      { foreignKey: 'medecin_id', as: 'services' });

module.exports = { sequelize, User, Service, Reservation };
```

### Comparaison avec Eloquent Laravel

| Eloquent Laravel | Sequelize Express |
|---|---|
| `$user->hasMany(Service::class, 'medecin_id')` | `User.hasMany(Service, { foreignKey: 'medecin_id' })` |
| `$service->belongsTo(User::class)` | `Service.belongsTo(User, { as: 'medecin' })` |
| `$service->load('medecin')` | `include: [{ model: User, as: 'medecin' }]` |

## 12.9 Étape 4 – Middleware d'authentification JWT

### 📁 `src/middleware/auth.js`

```js
const jwt = require('jsonwebtoken');
const { User } = require('../models');

const authenticate = async (req, res, next) => {
  try {
    // 1. Extraire le token de l'en-tête Authorization: Bearer <token>
    const authHeader = req.headers['authorization'];
    if (!authHeader || !authHeader.startsWith('Bearer ')) {
      return res.status(401).json({ message: 'Token non fourni.' });
    }
    const token = authHeader.split(' ')[1];

    // 2. Vérifier et décoder le token
    let decoded;
    try {
      decoded = jwt.verify(token, process.env.JWT_SECRET);
    } catch (err) {
      if (err.name === 'TokenExpiredError') {
        return res.status(401).json({ message: 'Token expiré. Reconnectez-vous.' });
      }
      return res.status(401).json({ message: 'Token invalide.' });
    }

    // 3. Charger l'utilisateur depuis la base
    const user = await User.findByPk(decoded.id);
    if (!user) return res.status(401).json({ message: 'Utilisateur introuvable.' });

    // 4. Attacher l'utilisateur à la requête
    req.user = user;
    next(); // passer au middleware/contrôleur suivant
  } catch (error) {
    return res.status(500).json({ message: 'Erreur serveur.' });
  }
};

module.exports = authenticate;
```

### Comparaison avec Laravel

| Laravel | Express |
|---|---|
| `middleware('auth:sanctum')` sur la route | `authenticate` dans le tableau de middlewares |
| `$request->user()` dans le controller | `req.user` dans le controller |
| Token dans header `Authorization: Bearer` | Identique |

## 12.10 Étape 5 – Middleware de rôle

### 📁 `src/middleware/role.js`

```js
const role = (...roles) => {
  return (req, res, next) => {
    if (!req.user) {
      return res.status(401).json({ message: 'Non authentifié.' });
    }
    if (!roles.includes(req.user.role)) {
      return res.status(403).json({
        message: `Accès interdit. Rôle requis : ${roles.join(' ou ')}.`,
      });
    }
    next();
  };
};

module.exports = role;
```

### Utilisation dans les routes

```js
// Un seul rôle
router.get('/admin/users', authenticate, role('admin'), adminListUsers);

// Plusieurs rôles acceptés
router.get('/dashboard', authenticate, role('admin', 'medecin'), dashboard);
```

### Comparaison avec Laravel

| Laravel | Express |
|---|---|
| `middleware('role:admin')` | `role('admin')` en middleware |
| `Gate::authorize('admin')` | `role('admin', 'medecin')` |

## 12.11 Étape 6 – Contrôleur Auth

### 📁 `src/controllers/auth.controller.js`

#### register

```js
const register = async (req, res) => {
  const errors = validationResult(req);
  if (!errors.isEmpty()) {
    return res.status(422).json({
      message: 'Erreur de validation.',
      errors: errors.mapped(),
    });
  }

  const { name, email, password, role } = req.body;

  const existing = await User.findOne({ where: { email } });
  if (existing) {
    return res.status(422).json({
      message: 'Erreur de validation.',
      errors: { email: { msg: 'Cet email est déjà utilisé.' } },
    });
  }

  const hashedPassword = await bcrypt.hash(password, 10);  // ← compatible Laravel

  const user = await User.create({ name, email, password: hashedPassword, role });
  const token = jwt.sign(
    { id: user.id, email: user.email, role: user.role },
    process.env.JWT_SECRET,
    { expiresIn: process.env.JWT_EXPIRES_IN }
  );

  return res.status(201).json({ message: 'Inscription réussie.', user, token });
};
```

---

#### login

```js
const login = async (req, res) => {
  const { email, password } = req.body;

  const user = await User.findOne({ where: { email } });
  if (!user) {
    return res.status(401).json({ message: 'Identifiants incorrects.' });
  }

  // bcrypt.compare est compatible avec les hash créés par Laravel Hash::make()
  const isMatch = await bcrypt.compare(password, user.password);
  if (!isMatch) {
    return res.status(401).json({ message: 'Identifiants incorrects.' });
  }

  const token = jwt.sign(
    { id: user.id, email: user.email, role: user.role },
    process.env.JWT_SECRET,
    { expiresIn: process.env.JWT_EXPIRES_IN }
  );

  return res.status(200).json({ message: 'Connexion réussie.', user, token });
};
```

> Remarque : `bcrypt.compare()` de bcryptjs est **compatible avec les hash produits par PHP's password_hash()** utilisé par Laravel. Les utilisateurs créés dans Laravel peuvent donc se connecter via Express.

---

#### logout et me

```js
// JWT est stateless : pas de session serveur à supprimer.
// On informe juste le client de supprimer son token.
const logout = async (req, res) => {
  return res.status(200).json({
    message: 'Déconnexion réussie. Supprimez le token côté client.',
  });
};

const me = async (req, res) => {
  return res.status(200).json({
    message: 'Profil récupéré.',
    user: {
      id: req.user.id,
      name: req.user.name,
      email: req.user.email,
      role: req.user.role,
    },
  });
};
```

### Différence importante : logout JWT vs Sanctum

| Laravel Sanctum | JWT Express |
|---|---|
| `$request->user()->currentAccessToken()->delete()` | Pas de suppression côté serveur |
| Token supprimé en base | Token reste valide jusqu'à expiration |
| Déconnexion immédiate | Déconnexion basée sur expiration |
| Pour invalidation réelle → blacklist Redis | |

## 12.13 Étape 8 – Contrôleur Réservations

### Créer une réservation (patient)

```js
const store = async (req, res) => {
  const errors = validationResult(req);
  if (!errors.isEmpty()) {
    return res.status(422).json({ message: 'Erreur de validation.', errors: errors.mapped() });
  }

  try {
    const { service_id, date_reservation, heure_reservation, commentaire } = req.body;

    // Vérifier que le service existe et est actif
    const service = await Service.findOne({ where: { id: service_id, statut: 'actif' } });
    if (!service) return res.status(404).json({ message: 'Service introuvable ou inactif.' });

    const reservation = await Reservation.create({
      user_id: req.user.id,    // le patient connecté
      service_id,
      date_reservation,
      heure_reservation,
      statut: 'en_attente',
      commentaire: commentaire || null,
    });

    return res.status(201).json({ message: 'Réservation créée.', data: reservation });
  } catch (error) {
    return res.status(500).json({ message: 'Erreur serveur.' });
  }
};
```

---

### Annuler une réservation (patient)

```js
const cancel = async (req, res) => {
  try {
    // Un patient ne peut annuler QUE ses propres réservations
    const reservation = await Reservation.findOne({
      where: { id: req.params.id, user_id: req.user.id },
    });
    if (!reservation) return res.status(404).json({ message: 'Réservation introuvable.' });

    if (reservation.statut === 'annulee') {
      return res.status(400).json({ message: 'Déjà annulée.' });
    }

    await reservation.update({ statut: 'annulee' });
    return res.status(200).json({ message: 'Réservation annulée.', data: reservation });
  } catch (error) {
    return res.status(500).json({ message: 'Erreur serveur.' });
  }
};
```

> Remarque : `req.user.id` est automatiquement disponible grâce au middleware `authenticate`. Le patient ne peut jamais annuler la réservation d'un autre utilisateur.

## 12.15 Étape 10 – Configuration Swagger

### Qu'est-ce que Swagger / OpenAPI ?

**Swagger** est un standard de documentation des API REST.

* Format : **OpenAPI 3.0** (fichier JSON/YAML décrivant toutes les routes)
* Interface : **Swagger UI** affiche une page web interactive pour tester l'API
* Avantage : la documentation est générée **depuis le code** (annotations JSDoc)

### Architecture Swagger

```
Routes JS          swagger-jsdoc         swagger-ui-express
(annotations)  →  (génère OpenAPI JSON)  →  (affiche l'UI web)
    @swagger          swaggerSpec            GET /api-docs
```

---

### 📁 `src/config/swagger.js`

```js
const swaggerJsdoc = require('swagger-jsdoc');

const options = {
  definition: {
    openapi: '3.0.0',
    info: {
      title: 'Gestion Med API',
      version: '1.0.0',
      description: 'API REST de réservation médicale — Backend Express.js',
    },
    servers: [{ url: 'http://localhost:3000', description: 'Développement' }],
    components: {
      securitySchemes: {
        bearerAuth: {
          type: 'http',
          scheme: 'bearer',
          bearerFormat: 'JWT',
          description: 'Entrez votre token JWT obtenu via POST /api/v1/login',
        },
      },
    },
    tags: [
      { name: 'Auth',     description: 'Inscription, connexion, déconnexion' },
      { name: 'Services', description: 'Consultation des services médicaux' },
      { name: 'Patient',  description: 'Gestion des réservations' },
      { name: 'Médecin',  description: 'Gestion des services et réservations' },
      { name: 'Admin',    description: 'Administration complète' },

```

## 12.16 Étape 11 – Annoter les routes avec @swagger

Les annotations Swagger se placent dans des commentaires JSDoc **directement au-dessus** de chaque route.

### Exemple : route POST /register

```js
/**
 * @swagger
 * /api/v1/register:
 *   post:
 *     summary: Inscription d'un nouvel utilisateur
 *     tags: [Auth]
 *     requestBody:
 *       required: true
 *       content:
 *         application/json:
 *           schema:
 *             type: object
 *             required: [name, email, password]
 *             properties:
 *               name:     { type: string, example: Jean Dupont }
 *               email:    { type: string, example: jean@example.com }
 *               password: { type: string, example: secret123 }
 *     responses:
 *       201:
 *         description: Inscription réussie
 *       422:
 *         description: Erreur de validation
 */
router.post('/register', [...validations], register);
```

---

### Exemple : route protégée (Bearer JWT)

```js
/**
 * @swagger
 * /api/v1/me:
 *   get:
 *     summary: Profil de l'utilisateur connecté
 *     tags: [Auth]
 *     security:
 *       - bearerAuth: []    ← indique que cette route nécessite un token
 *     responses:
 *       200:
 *         description: Profil récupéré
 *       401:
 *         description: Non authentifié
 */
router.get('/me', authenticate, me);
```

### Structure d'une annotation @swagger

```yaml
@swagger
/api/v1/chemin:     ← URL de l'endpoint
  methode:          ← get | post | put | delete
    summary:        ← description courte
    tags:           ← groupe dans l'UI
    security:       ← [] = public, bearerAuth = protégé
    parameters:     ← paramètres d'URL (:id)
    requestBody:    ← corps de la requête (POST/PUT)
    responses:      ← codes HTTP possibles
```

## 12.19 Récapitulatif des concepts Express/Node utilisés

### Architecture middleware d'Express

```
app.use(cors())           ← 1. Autoriser les origines cross-domain
app.use(express.json())   ← 2. Parser le body JSON
app.use('/api-docs', ...) ← 3. Swagger UI
app.use('/api/v1', ...)   ← 4. Toutes les routes API
app.use((req,res)=>{})    ← 5. Route 404 (catch-all)
app.use((err,req,res)=>{})← 6. Gestionnaire d'erreurs global
```

### Flux d'une requête authentifiée

```
Client → Header: Authorization: Bearer <token>
           │
           ▼
         auth.js      → jwt.verify() → req.user = User
           │
           ▼
         role.js      → req.user.role === 'admin' ?
           │
           ▼
         controller   → Sequelize query → JSON response
```

| Concept | Implémentation |
|---|---|
| Routing | express.Router() dans src/routes/ |
| Middlewares | fonctions (req, res, next) chainées |

| ORM | Sequelize avec `underscored: true` |

## 12.20 Erreurs fréquentes et solutions

### Erreur 1 : `SequelizeConnectionRefusedError`

```
Error: connect ECONNREFUSED 127.0.0.1:3306
```

**Cause** : MySQL n'est pas démarré.
**Solution** : Démarrer WAMP/XAMPP, vérifier que MySQL tourne.

---

### Erreur 2 : `JWT malformed` ou `invalid signature`

```json
{ "message": "Token invalide." }
```

**Cause** : Le `JWT_SECRET` dans `.env` ne correspond pas à celui utilisé pour générer le token.
**Solution** : Vérifier que `.env` est bien chargé, redémarrer le serveur après modification.

---

### Erreur 3 : `column does not exist` ou champs `null`

**Cause** : `underscored: true` manquant dans la définition du modèle.
**Solution** : S'assurer que chaque modèle a `underscored: true` dans ses options.

---

### Erreur 4 : `CORS Error` depuis Vue.js

```
Access to XMLHttpRequest blocked by CORS policy
```

**Cause** : L'origine `http://localhost:5173` n'est pas autorisée.
**Solution** : Vérifier la config CORS dans `server.js` :

```js
app.use(cors({
  origin: ['http://localhost:5173', 'http://127.0.0.1:5173'],
}));
```

---

### Erreur 5 : `403 Forbidden` alors que connecté

**Cause** : L'utilisateur est connecté mais son rôle ne correspond pas.
**Solution** : Vérifier le rôle de l'utilisateur (`GET /api/v1/me`), ou utiliser `POST /api/v1/admin/users/:id/set-role` pour changer le rôle.

---

### Erreur 6 : `422 Unprocessable Entity` sans message clair

**Cause** : Les validations express-validator ont échoué.
**Solution** : Lire le champ `errors` dans la réponse JSON pour identifier le champ invalide.

```json
{
  "message": "Erreur de validation.",
  "errors": {
    "email": { "msg": "Email invalide." }
  }
}
```

## 12.21 Aller plus loin

### 1. Middleware de validation centralisé

Au lieu de répéter `validationResult(req)` dans chaque controller, créer un middleware :

```js
// src/middleware/validate.js
const { validationResult } = require('express-validator');

const validate = (req, res, next) => {
  const errors = validationResult(req);
  if (!errors.isEmpty()) {
    return res.status(422).json({
      message: 'Erreur de validation.',
      errors: errors.mapped(),
    });
  }
  next();
};

module.exports = validate;
```

Utilisation dans les routes :

```js
router.post('/register', [...rules], validate, register);
```

### 2. Blacklist JWT avec Redis (logout réel)

Avec JWT stateless, le logout ne supprime pas le token du serveur.
Pour une invalidation réelle, on peut utiliser **Redis** comme blacklist :

```js
// À l'installation : npm install ioredis
const Redis = require('ioredis');
const redis = new Redis();

// Lors du logout
const logout = async (req, res) => {
  const token = req.headers['authorization'].split(' ')[1];
  const decoded = jwt.decode(token);
  const ttl = decoded.exp - Math.floor(Date.now() / 1000);
  await redis.setex(`blacklist:${token}`, ttl, '1');
  return res.json({ message: 'Déconnecté.' });
};

// Dans le middleware auth, vérifier la blacklist
const isBlacklisted = await redis.get(`blacklist:${token}`);
if (isBlacklisted) return res.status(401).json({ message: 'Token révoqué.' });
```
