---

# 2. Architecture logique des contrôleurs

Le projet suit une structure **par rôle**, ce qui est cohérent avec les exigences fonctionnelles :

| Rôle           | Accès autorisé                                      |
| -------------- | --------------------------------------------------- |
| Public         | Consultation des services                           |
| Patient        | Création & gestion des réservations personnelles    |
| Médecin        | Consultation des rendez-vous, mise à jour du statut |
| Administrateur | CRUD services, utilisateurs, réservations           |

Cette structure évite :

* les contrôleurs trop volumineux (anti-pattern "God Controller"),
* la confusion des responsabilités,
* les problèmes de sécurité liés à l’absence de séparation des droits.

**Conseil pédagogique :**
Insister sur l’importance de la séparation des responsabilités et leur impact sur la maintenabilité.

# 3. Analyse détaillée des contrôleurs

## 3.1 ServiceController

### Intérêt pédagogique :

* Introduire l’idée de contrôleur public.
* Utiliser Eloquent avec relations simples (`belongsTo`).
* Préparer les étudiants à la pagination plus tard.

### Points critiques :

* L’utilisation de `with('medecin')` permet de réduire drastiquement le nombre de requêtes SQL.
  Sans `with()`, Laravel ferait une requête par service (N+1 problem).

* `findOrFail` génère automatiquement une page 404 gérée par Laravel.
  Cela montre la puissance de la gestion d’erreurs intégrée.

### Défaut fréquent des étudiants :

* Oublient de filtrer les services inactifs
* Oublient `with('medecin')` → base sollicitée 10x plus
* Appellent la mauvaise vue ou l'oublient

---

## 3.2 DashboardController

### Intérêt pédagogique :

* Faire comprendre le concept d’accès dynamique selon rôle.
* Montrer comment utiliser `auth()->user()` en sécurité.
* Préparer le prochain chapitre : les middlewares.

### Erreurs fréquentes :

* Conditions mal écrites (`=` au lieu de `==` ou `===`).
* Mauvaise orthographe des rôles (`admin`, `admim`, etc.).
* Mauvais chemin de vue (fichier non créé).

### À surveiller :

* Vérifier que l’étudiant crée réellement les fichiers dans `resources/views/dashboard/`.

---

## 3.3 ReservationController (patient)

### Points clés :

* Nécessite que les étudiants comprennent la validation.
* Première fois qu’ils confrontent une logique métier complète :
  créer → afficher → annuler.

#### Validation

La validation côté contrôleur est essentielle pour éviter :

* les réservations dans le passé,
* les services inexistants,
* les utilisateurs qui contournent le formulaire avec des outils comme Postman.

#### Problèmes potentiels :

* Format de date incompatible avec MySQL
* Heures non valides
* Ressources protégées non gérées

### L’annulation

Important pédagogiquement :
Une réservation n’est annulable que si elle est encore “en attente”.

Les étudiants oublient souvent ce contrôle.

---

## 3.4 MedecinController

### Intérêt technique :

* Présente une requête avancée avec `whereHas()`
* Montre comment filtrer sur une relation, ce que les étudiants trouvent fascinant.

### Erreur fréquente :

* Les étudiants tentent d’écrire :

```php
Reservation::where('service.medecin_id', auth()->id())->get();
```

Ce qui est impossible en SQL.

Explique-leur que `whereHas()` est l’outil correct pour filtrer via des relations.

### Mise à jour du statut

Il faut absolument vérifier l’appartenance du service au médecin.

Les étudiants omettent souvent :

```php
if ($reservation->service->medecin_id != auth()->id()) { abort(403); }
```

et cela ouvre une faille grave (injection de statut).

# 4. Relation entre routes, contrôleurs et vues

Une erreur courante des étudiants :

* Créer le contrôleur mais oublier d’ajouter la route
* Appeler la vue `services.index.blade.php` alors qu'ils ont créé `services.blade.php`
* Se tromper dans le chemin de vue

**Conseil important à donner :**

Si Laravel affiche :
`View [services.index] not found.`
cela signifie que la vue n’est pas où elle devrait être :

```
resources/views/services/index.blade.php
```

# 5. Validation et sécurité (explication avancée)

### Pourquoi valider dans les contrôleurs ?

Parce qu’un étudiant ou un utilisateur malveillant peut contourner les formulaires HTML et envoyer :

* une date invalide
* un ID de service inexistant
* un statut non autorisé
* un user_id d’un autre patient

### La ligne la plus importante pour la sécurité :

```php
if ($reservation->user_id != auth()->id()) { abort(403); }
```

Cette vérification empêche une **usurpation de réservation**.

# 6. Eloquent ORM – ce que les étudiants ne comprennent pas tout de suite

Eloquent repose sur une idée :

Un modèle = une table

Un objet = une ligne dans la table

Exemples à rappeler régulièrement :

* `Service::all()` = toutes les lignes
* `Service::find($id)` = une seule ligne
* `$service->reservations` = toutes les réservations liées
* `$service->medecin` = médecin lié au service

L’ORM permet aux étudiants de manipuler la base **sans écrire une seule requête SQL brute**.

# 7. Les tests que tu dois faire pour valider leur travail

### 1. Services publics

* aller sur `/services`
* vérifier la liste
* afficher un service

### 2. Patient

* réserver un service
* vérifier `/mes-reservations`
* annuler
* essayer d'annuler deux fois (doit échouer)

### 3. Médecin

* consulter ses services
* consulter les réservations associées
* changer le statut
* vérifier l’impact sur l’interface patient

### 4. Administrateur

* créer un service
* modifier
* supprimer
* assigner un médecin
* changer un rôle utilisateur

# 8. Problèmes fréquents et solutions

### Erreur : “Undefined variable: service”

Cause : la variable n’a pas été envoyée dans `compact()`.

### Erreur : “Trying to get property of non-object”

Cause : `$service` est null → mauvais id → utiliser `findOrFail`.

### Erreur : “403 Forbidden”

Cause : utilisateur non autorisé → corriger middleware ou vérification du rôle.

### Erreur : “SQLSTATE[HY000]: General error: 1364 Field doesn't have a default value”

Cause : champ obligatoire manquant → corriger `$fillable` ou `validate()`.
