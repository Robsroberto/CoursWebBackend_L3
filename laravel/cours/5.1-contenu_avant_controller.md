## 1. Principe général d’une migration : `up()` et `down()`

Une migration est une classe PHP qui décrit **une modification de la base de données**.

* `up()` : décrit **ce qu’on fait** (créer une table, ajouter une colonne, etc.)
* `down()` : décrit **comment annuler** ce qu’on a fait dans `up()` (supprimer la table, enlever la colonne, etc.)

Laravel utilise ces deux méthodes pour :

* `php artisan migrate` → exécute `up()`
* `php artisan migrate:rollback` → exécute `down()` pour revenir en arrière

L’idée : chaque migration est réversible.

---

## 2. Migration `create_services_table` — explication détaillée

Migration simplifiée (partie importante) :

```php
public function up(): void
{
    Schema::create('services', function (Blueprint $table) {
        $table->id();
        $table->string('titre');
        $table->text('description')->nullable();
        $table->decimal('prix', 8, 2)->default(0);
        $table->integer('duree');
        $table->enum('statut', ['actif', 'inactif'])->default('actif');
        $table->unsignedBigInteger('medecin_id')->nullable();
        $table->timestamps();

        $table->foreign('medecin_id')->references('id')->on('users');
    });
}

public function down(): void
{
    Schema::dropIfExists('services');
}
```

### 2.1. Signature de `up()`

```php
public function up(): void
```

* `public` : visibilité → la méthode est accessible depuis l’extérieur (Laravel peut l’appeler).
* `function` : mot-clé pour déclarer une fonction / méthode.
* `up` : nom de la méthode → Laravel connaît ce nom, il ne faut pas le changer.
* `(): void` : type de retour → signifie “ne renvoie rien”.

### 2.2. `Schema::create(...)`

```php
Schema::create('services', function (Blueprint $table) {
    ...
});
```

* `Schema` : façade Laravel qui encapsule la logique de création/modification de tables SQL.
* `::` : opérateur de portée de classe (“résolution de méthode statique”).
* `create` : méthode statique qui dit “crée une nouvelle table”.
* `'services'` : nom de la table dans la base MySQL.
* `function (Blueprint $table) { ... }`

  * On passe une **fonction anonyme** (closure) en deuxième argument.
  * `Blueprint` est la classe qui contient les méthodes pour définir les colonnes.
  * `$table` est l’objet qui représente la table en cours de définition.
  * Tout ce qui est dedans (`$table->string(...)`, `$table->integer(...)`, etc.) ajoute des colonnes à cette table.

---

### 2.3. `$table->id();`

```php
$table->id();
```

* `$table` : l’objet Blueprint reçu dans la fonction anonyme.
* `->` : opérateur d’accès à une méthode ou propriété d’objet.
* `id()` : méthode qui indique :

  * crée une colonne `id` de type entier non signé auto-incrémenté,
  * et la définit comme **clé primaire** (`PRIMARY KEY`).

En SQL, cela devient grosso modo :

```sql
id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY
```

---

### 2.4. `$table->string('titre');`

```php
$table->string('titre');
```

* Crée une colonne :

  * nom : `titre`
  * type : `VARCHAR(255)` par défaut

Utilisation attendue : texte court comme un titre, un nom, etc.

---

### 2.5. `$table->text('description')->nullable();`

```php
$table->text('description')->nullable();
```

* `$table->text('description')` : crée une colonne `description` de type `TEXT` (contenu plus long).
* `->nullable()` : enchaînement de méthodes (fluent interface).

  * Permet les valeurs `NULL` dans cette colonne (champ facultatif).

En SQL : `description TEXT NULL`.

---

### 2.6. `$table->decimal('prix', 8, 2)->default(0);`

```php
$table->decimal('prix', 8, 2)->default(0);
```

* `decimal('prix', 8, 2)` :

  * type DECIMAL avec précision :

    * `8` : nombre total de chiffres (ex : 123456.78 → 8 chiffres)
    * `2` : nombre de chiffres après la virgule
* `prix` : nom de la colonne
* `->default(0)` : valeur par défaut si rien n’est fourni → 0.00

En SQL : `prix DECIMAL(8,2) NOT NULL DEFAULT 0`.

---

### 2.7. `$table->integer('duree');`

```php
$table->integer('duree');
```

* Crée une colonne `duree` de type `INT`.
* Utilisée pour représenter un nombre entier (par exemple, durée en minutes).

---

### 2.8. `$table->enum('statut', ['actif', 'inactif'])->default('actif');`

```php
$table->enum('statut', ['actif', 'inactif'])->default('actif');
```

* `enum` : type qui limite les valeurs possibles.
* `'statut'` : nom de la colonne.
* `['actif', 'inactif']` : ensemble des valeurs acceptées.
* `->default('actif')` : si rien n’est précisé, la valeur sera “actif”.

En SQL : `statut ENUM('actif','inactif') DEFAULT 'actif'`.

---

### 2.9. `$table->unsignedBigInteger('medecin_id')->nullable();`

```php
$table->unsignedBigInteger('medecin_id')->nullable();
```

* `unsignedBigInteger` : entier long non signé (correspond à ce que crée `id()` dans users).
* `medecin_id` : clé étrangère qui va pointer vers `users.id`.
* `->nullable()` : un service peut exister sans être assigné à un médecin précis.

---

### 2.10. `$table->timestamps();`

```php
$table->timestamps();
```

* Ajoute deux colonnes :

  * `created_at` (date de création de l’enregistrement)
  * `updated_at` (date de dernière mise à jour)
* Laravel les gère automatiquement si `public $timestamps = true;` ou par défaut dans le modèle.

---

### 2.11. Définition de la clé étrangère

```php
$table->foreign('medecin_id')->references('id')->on('users');
```

Lecture en français :

* `$table->foreign('medecin_id')` : on déclare que `medecin_id` est une **clé étrangère**.
* `->references('id')` : elle référence la colonne `id`.
* `->on('users')` : dans la table `users`.

En SQL :

```sql
FOREIGN KEY (medecin_id) REFERENCES users(id)
```

Cela garantit que `medecin_id` contient toujours un `id` existant dans `users` (ou NULL).

---

### 2.12. Méthode `down()` – rollback

```php
public function down(): void
{
    Schema::dropIfExists('services');
}
```

* `dropIfExists('services')` : supprime la table `services` si elle existe.
* C’est ce qui sera exécuté lors d’un `php artisan migrate:rollback`.

---

### 2.13. Même logique pour `reservations`

La migration `create_reservations_table` utilise les mêmes idées :

* `unsignedBigInteger('user_id')` + foreign key vers `users`
* `unsignedBigInteger('service_id')` + foreign key vers `services`
* `date('date_reservation')`, `time('heure_reservation')`
* `enum('statut', [...])` avec valeurs comme `en_attente`, `confirmee`, etc.

Tout ce qu’on a expliqué sur :

* `Schema::create`
* `$table->...`
* `foreign`
* `enum`, `date`, `time`, etc.

s’applique de la même manière.

## 3. Modèle `Service` — explication détaillée

Le modèle représente une table de la base sous forme d’objet.

### Code du modèle :

```php
namespace App\Models;

use Illuminate\Database\Eloquent\Factories\HasFactory;
use Illuminate\Database\Eloquent\Model;

class Service extends Model
{
    use HasFactory;

    protected $fillable = [
        'titre',
        'description',
        'prix',
        'duree',
        'statut',
        'medecin_id'
    ];

    public function medecin()
    {
        return $this->belongsTo(User::class, 'medecin_id');
    }

    public function reservations()
    {
        return $this->hasMany(Reservation::class);
    }
}
```

---

### 3.1. `namespace App\Models;`

```php
namespace App\Models;
```

* Le **namespace** est comme un “dossier logique” pour les classes PHP.
* Ici, la classe `Service` appartient à l’espace de noms `App\Models`.
* Cela évite les collisions de noms entre classes portant le même nom dans des parties différentes du projet.

---

### 3.2. `use ...`

```php
use Illuminate\Database\Eloquent\Factories\HasFactory;
use Illuminate\Database\Eloquent\Model;
```

* `use` : permet d’indiquer qu’on va utiliser ces classes avec leur nom court.
* `HasFactory` : trait utilisé par Eloquent pour les **factories** (génération d’enregistrements de test).
* `Model` : classe de base d’Eloquent, que tous les modèles doivent étendre.

---

### 3.3. Déclaration de la classe

```php
class Service extends Model
{
    ...
}
```

* `class Service` : définition de la classe Service.
* `extends Model` : Service hérite de toutes les fonctionnalités de `Illuminate\Database\Eloquent\Model` :

  * accès BD,
  * `create`, `find`, `where`, etc.

Grâce à `extends Model`, on peut faire :

```php
Service::all();
Service::find(1);
Service::create([...]);
```

---

### 3.4. `use HasFactory;`

```php
use HasFactory;
```

* `use` (ici, à l’intérieur de la classe) : inclusion d’un **trait** PHP.
* Un trait est un “morceau de code” réutilisable dans plusieurs classes.
* `HasFactory` ajoute la méthode `factory()` qui permet d’utiliser les factories Eloquent pour générer des données de test.

---

### 3.5. Propriété `$fillable`

```php
protected $fillable = [
    'titre',
    'description',
    'prix',
    'duree',
    'statut',
    'medecin_id'
];
```

* `protected` : visibilité → accessible dans la classe et ses classes filles, pas directement de l’extérieur.
* `$fillable` : tableau qui indique quelles colonnes peuvent être remplies de façon **massive** (`mass assignment`).

Exemple :

```php
Service::create([
    'titre' => 'Consultation',
    'description' => '...',
    'prix' => 10000,
    'duree' => 30,
    'statut' => 'actif',
    'medecin_id' => 2,
]);
```

Si un champs n’est pas dans `$fillable` :

* Laravel refuse l’assignation
* et déclenche une `MassAssignmentException`.

Sécurité : empêche qu’on remplisse des colonnes sensibles (role, is_admin, password, etc.) avec n’importe quel formulaire.

---

### 3.6. Relation `medecin()` → `belongsTo`

```php
public function medecin()
{
    return $this->belongsTo(User::class, 'medecin_id');
}
```

* `public function medecin()` :

  * définit une relation accessible comme `$service->medecin`.
* `return $this->belongsTo(...)` :

  * `belongsTo` indique que la table `services` contient une clé étrangère qui pointe vers `users`.
* `User::class` :

  * référence vers la classe `App\Models\User` (puisqu’on est dans le namespace `App\Models`).
* `'medecin_id'` :

  * nom de la clé étrangère dans la table `services`.

Utilisation :

```php
$service = Service::find(1);
$medecin = $service->medecin; // objet User
echo $medecin->name;
```

---

### 3.7. Relation `reservations()` → `hasMany`

```php
public function reservations()
{
    return $this->hasMany(Reservation::class);
}
```

* `hasMany` signifie : un service peut avoir **plusieurs** réservations.
* Laravel va chercher toutes les lignes de la table `reservations` où `service_id = id` du service.

Utilisation :

```php
$service = Service::find(1);
$reservations = $service->reservations; // collection d'objets Reservation
```

---

### 3.8. Modèle `Reservation` et `User` (même logique)

* `Reservation` :

  * `belongsTo(User::class)`
  * `belongsTo(Service::class)`

* `User` :

  * `hasMany(Reservation::class)`
  * `hasMany(Service::class, 'medecin_id')`

La logique est la même :

* `hasMany` → côté “parent” (un médecin, plusieurs services)
* `belongsTo` → côté “enfant” (un service, un médecin)
